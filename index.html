React Edition · Case Study

焼肉 黒耀 — React Reservation Experience

Vanilla JS 版の予約サイトを React / TypeScript で再構築した発展実装

なぜ React 化したか

予約は「日付 → 人数 → 時間 → 入力」と状態が増えるほど DOM 操作が複雑化します。Vanilla 版は DOM とグローバル変数で状態を持っていたため変更箇所が散らばり、表示ズレやバグが起きやすい構造でした。Reactなら状態を変えるだけで UI が自動で追従し、手動更新のバグを根本から減らせます。さらに TypeScript で予約データの「形」を型として定義し、受け渡しミスを実行前に検知できるようにしました。

使用技術

  • React 18
  • TypeScript
  • Vite 5
  • SCSS Modules
  • React Router
  • Context API

Vanilla JS 版との違い

状態管理DOM＋グローバル変数に分散→useState / Context で単一の源に集約
画面更新getElementById / innerHTML で手動→状態変更で UI が自動再描画
スタイルグローバルCSSで衝突リスク→SCSS Modules でスコープ分離
再利用コピーで対応→コンポーネント化して複数画面で共用
型安全なし（実行時に発覚）→型で事前に検知

コンポーネント分割

責務でフォルダを分け、画面を小さな部品の組み合わせで構築。CourseCard はトップとコース一覧で共用し、コースデータは1ソースで一元管理しています。

components/
├── ui/        … Button / SectionHeading / Reveal（汎用）
├── layout/    … Header / Footer / Layout
├── home/      … Hero / Concept / CoursePreview / ReserveCta
├── course/    … CourseCard（複数画面で再利用）
└── reserve/   … Calendar / GuestSelector / TimeSelector /
                 CustomerForm / ReservationSummary / StepIndicator

状態管理（State Management）

予約は「入力 → 確認 → 完了」とページをまたぐため、選択内容を React Context に集約。各ページは useReservation() で読み書きするだけで状態を共有できます。入力進捗（%）や必須項目チェックも Context 側で算出し、UI は状態を映すだけのシンプルな構造にしました。

Reservation UI の改善

「席を選ぶ体験」として、日付カレンダー（空席を ○△× 表示）→ 人数 → 時間帯 → お客様情報 の段階提示に再設計。右側の sticky サマリと進捗バーで「あと何を選べばいいか」を常に可視化しました。日付変更で時間を選び直す等の整合性も状態側で自動的に担保しています。

再利用性・保守性で意識した点

  • 再利用：CourseCard を複数画面で共用、データは1ソース。
  • 保守：責務でフォルダ分割＋型で、半年後でも追いやすい構造。
  • 拡張：予約APIの差し替え・多言語化・テーマ変更も影響範囲を局所化。
